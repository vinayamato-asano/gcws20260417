# ポモドーロタイマーWebアプリ アーキテクチャ案

## 1. 概要
Flask（Python）とHTML/CSS/JavaScriptを用いたシンプルかつ拡張性の高いWebアプリケーション構成を採用します。タイマーのロジックは主にフロントエンドで実装し、バックエンドは静的ファイル配信と必要なAPIのみを担当します。

---

## 2. ディレクトリ構成例

```
1.pomodoro/
  app.py                  # Flaskアプリ本体
  services/               # バックエンドロジック（テスト容易性向上のため分離）
    timer_service.py      # タイマー関連ロジック
    settings_service.py   # 設定管理ロジック
  tests/                  # バックエンド用ユニットテスト
    test_timer_service.py
    test_settings_service.py
  templates/
    index.html            # メイン画面
  static/
    js/
      timer.js           # タイマーロジック（モジュール化）
      api.js             # API通信
      ui.js              # UI操作
    tests/
      timer.test.js      # タイマーロジックのテスト
    css/
      style.css
```

---

## 3. バックエンド（Flask）
- 静的ファイル（HTML, CSS, JS, 画像）の配信
- 必要に応じてAPIエンドポイントの提供（設定保存・履歴取得など）
- ルーティングやAPI処理とビジネスロジックを分離し、テスト容易性を高める
- 依存性注入を意識し、DBアクセスや設定値取得をモックしやすくする
- テスト用設定ファイルを用意し、テスト時はテストDBやモックサービスを利用

---

## 4. フロントエンド（HTML/CSS/JavaScript）
- タイマーUIの描画・操作（index.html, style.css, ui.js）
- タイマーの状態管理・カウントダウン（timer.jsでクラスや関数として実装）
- API通信処理の分離（api.js）
- テストフレームワーク（Jest等）を導入し、ロジック部分のユニットテストを実装

---

## 5. テスト容易性のための工夫
- バックエンド・フロントエンドともに「ロジックとUI/ルーティングの分離」を徹底
- テストしやすいように依存性注入やモジュール化を意識
- テスト用ディレクトリ・設定を用意

 - 各開発ステップごとに、追加・変更されたロジックに対してユニットテストを実装・実行する運用とする
 - テストコードはバックエンド・フロントエンドともに専用ディレクトリに整理

---

## 6. 拡張性
- ユーザーごとの設定や履歴保存が必要な場合は、FlaskでDB（例：SQLite）を利用
- ログイン機能を追加する場合はFlask-Login等の導入を検討

---

## まとめ
- タイマーのロジックはフロントエンドで実装し、Flaskは静的ファイル配信＋必要なAPIのみ担当
- 機能追加時も拡張しやすい構成
- テスト容易性・保守性・拡張性を重視した設計
